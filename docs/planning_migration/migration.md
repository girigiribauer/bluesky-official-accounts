# モデレーションシステムへの段階的な移行計画

## 概要

旧カテゴリーシステムから新しい『分野別モデレーションシステム』への移行計画。
段階的なアプローチにより、リスクを最小化しながら、コミュニティ主導の『分野別モデレーションシステム』へ移行する。

## フェーズごとの計画

### フェーズ1: 投稿フォーム見直しとフロントエンド刷新（済）

協力者を今後より多く求めていく必要があるため、
現在のものが有用であると再認識してもらう必要がある。

そこで、一旦 Notion での管理はそのままに、
そこに至るまでの投稿フォームの自前化（Notion API を叩きつつバリデーションする）を行う。
また、同時に企業・組織向けに新規ページを追加しつつフロントエンドを刷新して、同時に協力者を募る。

### フェーズ2: モデレーションシステムの要件定義

新しいシステムの設計方針を固め、実装に必要な要件を定義する。

- 分野の確定
- モデレーションシステムの仕様の確定
- アーキテクチャ構成の設計
- モデリング
- UI/UX・画面設計

### フェーズ3: データ移行と管理者専用レビュー画面の構築（済）

Notion を捨てて自前の DB で運用できる状態にする。
管理者だけがアクセスできる画面を `/moderation_beta` に作り、Notion で行っていたレビュー作業を置き換える。
**表側（閲覧ページ）は一切変えない。**（ただし未移行（未確認）を未移行・未確認に分離することを除く）

- `/moderation_beta` に管理者専用の認証・レビュー画面を構築
  - Bluesky OAuth 認証（管理者のみ通過）
  - レビュー・承認・却下・編集の実装
- Notion から Supabase へのデータ移行
  - データベーススキーマの実装
  - Notion からの一括インポートスクリプト
- フォームの投稿先を Supabase に切り替え
  - 新旧分類を両方入力してもらう形式を維持
  - 表側・更新情報は変えない
- 管理者による運用開始・動作検証

#### DB設計の方針

スキーマの定義は `supabase/migrations/20260401000000_initial_schema.sql` を参照。

- `field_id` は `fields` テーブルで FK 管理。表示ラベルはコード定数 `FIELD_ID_LABELS` と一致させる
- `old_category` は移行期間中のみ使用。Notion からの移行時に新分野へ変換し、変換不可は未分類扱い
- 匿名投稿を許容する（公開フォームからの投稿は `submitted_by = null`）
- フォーム投稿は `entry_submissions` / `request_submissions` に一時保存し、モデレーション後に `entries` / `requests` として公開する
- `requests` と `entries` は別テーブルで管理。Bluesky アカウントが確認されたとき `requests.entry_id` を紐付けることで継続性を保つ
- `entries.transition_status` の `not_migrated` は Notion の「未移行（未確認）」に対応する移行専用値。Bluesky アカウントはあるが X との同一性が未確認の状態。新規申請では使用しない

**未決定事項:** `entry_submissions` のレビュー中に追記する根拠・操作ログをどこに保存するか。承認前は `account_id` が存在しないため `evidences` / `activities` に記録できない。

### フェーズ4: 新しい分野ベースへの切り替えと招待準備

新しい分野・分類ベースへの切り替えを完了させ、信頼できる少数の人を招く準備が一通り整った状態にする。

- 公開ページの表示を新分野・分類に切り替え
  - `fetchAccountList.ts` を `account_fields` / `fields` / `classifications` ベースに更新
  - `Account` モデルおよび公開ページのグルーピングを新分野ベースに変更
- オンボーディングの実装
  - ログイン後の分野選択フロー（`moderators` + `field_memberships` の登録）

### フェーズ5: 一般開放

管理者が介在しなくても運用が回る仕組みを整えたうえで、一般開放する。

- 段階制度の実装（3段階）
- ポイント制度・見回りタスク等のゲーミフィケーション
- 通知システム（サイト内通知のみ）
- 異議申し立て機能
- 旧分類の廃止（表・フォームから旧分類に関する記述をすべて削除、`old_category` カラム削除を含む）
- ログイン一般開放

### フェーズ6（将来）: atproto への対応

承認済みエントリーを `@official-accounts.bsky.social` の PDS に公開レコードとして保存し、公開サイトのデータソースを PDS に切り替える。

- `@official-accounts.bsky.social` の DID でレキシコンを設計・実装
- 承認時に PDS へのレコード書き込みを追加
- 既存の承認済みエントリーを PDS に移行するスクリプトの作成
- 公開サイトを PDS から読む形に切り替え
