# 振り返り：Notion → Supabase 移行インシデント（2026年4月）

## 概要

Notion から Supabase へのデータ移行・フォーム切り替えを実施した際、設計上の矛盾により本番フォームが10数分にわたって機能しない状態になった。Notion への切り戻しで安定化したが、設計を見直してから再移行が必要な状態である。

## 影響

- 登録フォームから送信すると「送信に失敗しました」と表示され、投稿が一切保存されなかった
- 影響を受けたのは既存アカウントを送信しようとしたユーザー（新規アカウントは正常に送信できた可能性がある）
- 発生時間：4月29日のデプロイから切り戻し完了まで、10数分程度

## タイムライン

- **4月19日**: フォーム投稿先・fetch スクリプトを Supabase に切り替えた commit を作成（本番未反映）
- **4月28〜29日**: Supabase へのデータ移行実行・GitHub Actions 切り替え・本番 push
- **4月29日**: フォームからテスト投稿 → 既存アカウントで `duplicate key` エラー発覚
- **4月29日**: Notion への切り戻し完了・動作確認

## 根本原因

### 設計上の矛盾

Notion では「申請」と「公開済み」が同じデータベースに複数行として共存できる。同じアカウントへの修正投稿も新しい行として追加され、モデレーターがレビューして公開済み行を更新する運用だった。

Supabase のスキーマでは `entries` テーブルに 1 DID = 1行を前提とした設計（`bluesky_did` に UNIQUE 制約）にしたため、既存アカウントへの再申請・修正投稿が一切できない状態になっていた。フォームの設計意図（既存アカウントへの修正投稿も受け付ける）を正しく理解しないまま、また Notion との運用の差異を洗い出さないままスキーマを設計したことが直接の原因。

`submissions`（申請）と `entries`（確定済み）を分離すべきだったが、同一テーブルで `status` フィールドで管理しようとしたことで矛盾が生じた。

### プロセス上の問題

- **計画なしで作業を積み重ねた**: 残タスクの把握・共有がなく、何が終わって何が残っているかをその都度確認できない状態が続いた
- **デプロイ前の E2E 検証をしなかった**: フォーム送信 → DB 保存 → モデレーション画面確認という一連の流れを本番前に通していなかった
- **ロールバック手順を用意していなかった**: 問題が起きてから初めて「どう戻すか」を考えた
- **影響範囲を説明せず実行した**: 作業の依存関係や副作用を整理しないまま進めた
- **スクリプトの要否判断を誤った**: `migrateNewsFromNotion.ts` を未実行のまま「不要」と判断し削除を促した（ファイルが残っていたため事なきを得た）
- **状況把握を繰り返し誤った**: git の push タイミング、categories.json の用途、フォームの設計意図など、複数の場面で誤った理解のまま作業を進めた

## 対応内容

- `git revert` で6コミット分を切り戻し、Notion 運用に復帰した
- フォームからの投稿が Notion に正常に届くことを確認した

## 再発防止策

### 再移行前にやること

1. スキーマの設計見直し
   - `submissions` テーブルを新設し、フォーム申請はすべて `submissions` に INSERT する
   - `entries` は「確定済み公開アカウント」のみを持つテーブルとして再定義する
   - モデレーターが `submissions` を承認したとき、初回なら `entries` に INSERT、更新なら UPDATE する

2. フォームの動作を設計通りに検証する
   - 新規アカウントの申請
   - 既存アカウントへの修正申請
   - 来て欲しいフォームの動作確認

3. ロールバック手順を事前に明文化する

### プロセスの改善

- **着手前に計画を立てる**: 残タスクと依存関係を一覧化し、作業の順序と影響範囲を合意してから着手する
- 本番デプロイ前に一連のフローを通しで検証する
- 設計変更を伴う作業は、既存の運用との差異を先に洗い出してからスキーマを決める
